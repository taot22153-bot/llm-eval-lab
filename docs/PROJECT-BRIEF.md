# LLM Eval Lab 项目简报

## 目标

为大模型应用开发者提供本地优先、可重复、可解释的质量与安全评测，帮助用户判断候选应用版本能否替代基准版本并发布。

## 目标用户

开发大模型应用的个人开发者或小团队。第一版交付形态为本机单用户应用，不包含注册、登录和多人协作。

## 演示场景

平台内置一个虚构数码商店的电商智能客服作为样例应用。演示数据全部自行编写，包括商品、配送、退换货和保修规则，以及正常、幻觉诱导、提示注入和越狱测试用例。

## 核心流程

1. 创建由模型、系统提示词、参数及可选知识库或工具配置组成的应用版本。
2. 选定一个基准版本和一个候选版本。
3. 使用同一版本化评测集同时运行两个版本。
4. 通过确定性规则和本地评分模型进行分层评分。
5. 将评分冲突、证据不足或低置信度结果送入人工复核。
6. 比较质量、安全性、延迟、成本和回归用例。
7. 输出通过、未通过或需要人工复核的发布判定。

## 测试用例

每条测试用例包含用户输入、应用应依据的背景资料、必须出现的正确事实、禁止出现的错误或敏感信息、测试类型、严重级别和人工复核要求。第一版支持正常问答、幻觉诱导和安全攻击三类测试，严重级别分为普通、重要和阻断发布。

## 发布规则

- 任意阻断发布用例失败时，候选版本未通过。
- 候选版本相对基准版本新增重要错误时，候选版本未通过。
- 自动评分冲突、证据不足或置信度较低时，需要人工复核。
- 正确率不得下降，且安全测试必须全部通过。
- 延迟和成本不得超过用户设置的预算。
- 默认阈值可由用户调整，发布规则本身需要版本化。

## 约束

- 每周投入 4-6 小时。
- 6 周完成可演示版本，10 周完成面试版本。
- 核心流程必须在用户的 Windows 电脑上离线运行。
- 云模型 API 和租用 GPU 只能作为可插拔增强能力。
- 默认从恢复 Python 和调用现成模型开始，不要求预先掌握大模型训练。

## 第一版不做

- 注册、登录、权限和多人协作。
- 大模型预训练、RLHF 和大规模微调。
- 通用模型排行榜。
- 依赖云服务才能完成的核心评测。

## 公开与协作

- 项目最终发布为公开 GitHub 仓库。
- 每个完成的独立任务都要提交并 push 到 GitHub，功能任务通过 Issue、任务分支和 Pull Request 留下完整记录。
- 只有通过验收、测试和本地验证的功能才标记为完成。
- 仓库只提供 `.env.example`，不提交真实 API 密钥、数据库凭据、模型文件或运行数据。

## 交付形态

第一版为运行在用户 Windows 电脑上的本地 Web 控制台。用户通过浏览器创建和比较应用版本、启动评测、查看进度与指标、检查失败案例、完成人工复核并查看发布判定。

## 第一版验收

项目必须能够稳定完成一次不超过 5 分钟的本地面试演示：打开 Web 控制台，选择电商客服的基准版本和候选版本，运行内置评测集，展示实时进度和版本指标，定位候选版本新增的幻觉或安全问题，人工复核一条冲突结果，并输出带具体原因的发布判定。

## 模型入口

- 通过 Ollama 接入 Windows 本地模型，确保核心评测可以离线完成。
- 通过 OpenAI 兼容接口接入云模型或租用 GPU 服务。
- 两种入口共用统一模型适配边界，评测流程不依赖具体模型厂商。
- 第一版不为不同厂商分别实现专用集成，也不预先锁定具体模型。

## 数据存储

- 使用 MySQL 作为主数据库，保存应用版本、评测集、测试用例、运行记录、自动评分、人工复核和发布判定。
- MySQL Workbench 仅作为可选的图形化管理工具，不是应用运行依赖。
- 当前电脑尚未安装 MySQL Server 或 Workbench，项目启动阶段再配置本地服务或 Docker 启动方式。
