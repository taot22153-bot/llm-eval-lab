# 开发与 GitHub 同步流程

## 任务单位

每个任务应是一段可独立验收的纵向功能，必须包含明确的完成条件。仅创建文件、写出部分代码或留下无法运行的中间状态不算完成。

## 标准流程

1. 为任务创建 GitHub Issue，写明目标、范围和验收条件。
2. 从最新主分支创建任务分支，例如 `feat/12-run-evaluation`。
3. 在本地实现功能并补充必要测试和文档。
4. 运行与任务范围相匹配的检查，确认验收条件全部满足。
5. 使用清晰的 Conventional Commit 提交，例如 `feat: add evaluation run creation`。
6. 将任务分支 push 到 GitHub，并创建 Pull Request。
7. 在 PR 中记录实现内容、验证结果、截图或演示证据和已知限制。
8. 审查通过且持续集成检查成功后合并到主分支。

## 完成定义

一个任务同时满足以下条件才标记为完成：

- 验收条件已满足。
- 相关自动测试通过。
- 本地关键流程已实际运行验证。
- 没有提交密钥、模型文件、数据库数据或其他敏感内容。
- 文档与实际行为一致。
- 任务分支已经 push，Pull Request 留有可核验记录。

## 例外

纯拼写修正等低风险机械改动可以直接提交主分支；功能、数据模型、接口和评测逻辑变更必须走任务分支和 Pull Request。
